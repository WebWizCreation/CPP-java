③ EXACT JSON SCHEMA — REPLICATE THIS PRECISELY
================================================================================

Every JSON file has this top-level shape:

{
  "id": "CPP_Ch3_Functions_Recursion",
  "track": "cpp",
  "chapter": 3,
  "title": "Functions & Recursion",
  "description": "One compelling sentence describing this chapter's purpose and value.",
  "sections": [ ...array of section nodes... ]
}

------------------------------------------------------------
ALL VALID SECTION NODE TYPES:
------------------------------------------------------------

1. HEADING — Creates an h2/h3/h4 with right-sidebar TOC entry
   { "type": "heading", "level": 2, "content": "🔢 Section Title Here" }

   level 2 = major section header (use most often)
   level 3 = sub-section header
   level 4 = minor sub-section header


2. PARAGRAPH — Body text. Supports **bold**, *italic*, and `backtick inline code`
   { "type": "paragraph", "content": "Text with **bold** and `inline code` supported." }


3. CODE — Syntax-highlighted code block with copy button
   { "type": "code", "language": "cpp", "title": "Descriptive Title", "code": "// actual code here\nint x = 5;" }

   IMPORTANT: language must be exactly "cpp" or "java" (not "c++" or "Java").
   Use \n for newlines and \" for quotes inside the code string.


4. CALLOUT — Highlighted notice box
   { "type": "callout", "variant": "tip", "label": "Pro Tip Label", "content": "Body text with **bold** and `code` supported." }

   variant must be one of: "note" | "tip" | "warning" | "danger"
   Icons assigned automatically: note=📌  tip=💡  warning=⚠️  danger=🚨


5. TABLE — Data table
   {
     "type": "table",
     "headers": ["Col A", "Col B", "Col C"],
     "rows": [
       ["row1a", "row1b", "row1c"],
       ["`code in cell`", "**bold in cell**", "plain text"]
     ]
   }


6. LIST — Bullet or numbered list. Items support inline markdown.
   { "type": "list", "ordered": false, "items": ["**Item one** with detail", "`code item` explanation", "Plain item"] }
   Set "ordered": true for a numbered list.


7. DIVIDER — Horizontal rule separator (use sparingly, only before exercise sections)
   { "type": "divider" }


8. ACCORDION — Interactive exercise. Three variants:

   VARIANT A — "predict-output" (Guess what the program prints):
   {
     "type": "accordion",
     "variant": "predict-output",
     "title": "Exercise N: Description of what makes this tricky",
     "language": "java",
     "code": "public class Q1 {\n    public static void main(String[] args) {\n        // code here\n    }\n}",
     "answer": "exact\noutput\nhere",
     "explanation": "Line-by-line trace explaining exactly why each line prints what it does."
   }

   VARIANT B — "find-error" (Spot and fix the bug):
   {
     "type": "accordion",
     "variant": "find-error",
     "title": "Exercise N: Description of the buggy code",
     "language": "java",
     "code": "// buggy code here",
     "error": "Line X: Description of the exact error and its type (compile-time/runtime/logical)",
     "fix": "// corrected version of the code",
     "explanation": "Why this was wrong, what rule was violated, and how the fix resolves it."
   }

   VARIANT C — "problem-solving" (Implement a solution):
   {
     "type": "accordion",
     "variant": "problem-solving",
     "title": "Exercise N: Problem statement title",
     "language": "cpp",
     "code": "// Problem spec as comments — constraints, examples",
     "approach": "Step-by-step reasoning: what algorithm/pattern to use and why.",
     "solution": "// full working solution code",
     "explanation": "Why this approach is correct, complexity analysis, edge cases handled."
   }
